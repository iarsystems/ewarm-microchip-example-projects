00018  *

00024  * \section install Description
00025  * This embedded application source code illustrates how to implement a USB interfaced application
00026  * on the AVR32 microcontroller.
00027  *
00028  * As the AVR32 implements a device/host USB controller, the embedded application can operate
00029  * in one of the following USB operating modes:
00030  *   - USB device;
00031  *   - USB reduced-host controller;
00032  *   - USB dual-role device (depending on the ID pin).
00033  *
00034  * To optimize embedded code/RAM size and reduce the number of source modules, the application can be
00035  * configured to use one and only one of these operating modes.
00036  *
00037  * \section sample About the Sample Application
00038  * By default the sample code is delivered with a simple preconfigured dual-role USB application.
00039  * It means that the code generated allows to operate as a device or a host depending on the USB ID pin:
00040  *   - attached to a mini-B plug (ID pin unconnected) the application will be used in the device operating mode;
00041  *   - attached to a mini-A plug (ID pin tied to ground) the application will operate in reduced-host mode.
00042  *
00043  * \image html appli.jpg "Application Overview"
00044  *
00045  * Thus two instances of this application can be connected together with a mini-A - mini-B cable.
00046  *   - The host operating mode of the application (that we call A-device) can manage the connection
00047  * and the enumeration of a device application connected in device operating mode (that we call B-device).
00048  * Once the device has been enumerated, the host high-level application can operate USB applicative exchanges
00049  * with the B-device. Here the sample host application writes 64 bytes through a USB OUT pipe and reads back
00050  * this data with an IN pipe.
00051  *   - The device operating mode of the application (B-device) answers to the enumeration requests (setup requests)
00052  * of the host controller. Once it has been properly enumerated the high-level device application receives 64 bytes
00053  * through its OUT enpoint and when the host controller request for an IN exchange on its IN endpoints,
00054  * sends it back (loopback application).
00055  *
00056  * \note
00057  *    - When an EVK1101/UC3C-EK/EVK1104/EVK1105 is the host device, LED2 and LED3 will quickly blink. LED0 & LED1 will slowly blink with
00058  *    the following pattern: LED0 on/LED1 off, then LED0 off/LED1 on, then LED0 on/LED1 on.
00059  *    - When an EVK1100 is the host device, LED4(red light of the led tagged as LED5 on the PCB) will quickly blink.
00060  *    LED0, LED1, LED2 and LED3 will slowly blink in a complex pattern.
00061  *    - The host device may display the following messages on USART1(EVK1100/EVK1101)/USART0-CDC(EVK1105)/USART1-CDC(EVK1104)/USART2-CDC(UC3C-EK) depending
00062  *    on the current USB events: "Device connected", "Unsupported device",
00063  *    "Device enumerated", "USB suspended", "USB resumed". While the slave device
00064  *    never displays any message on USART.
00065  *
00066  * \section arch Architecture
00067  * As illustrated in the figure below, the application entry point is located in the enum_example.c file.
00068  * The main function first performs the initialization of services and tasks and then runs them in an infinite loop.
00069  *
00070  * The sample dual-role application is based on three different tasks:
00071  *   - The USB task (usb_task.c associated source file) is the task performing the USB low-level
00072  * enumeration process in device or host mode. Once this task has detected that the USB connection is fully
00073  * operational, it updates various status flags that can be checked within the high-level applicative tasks.
00074  *   - The device template task (device_template_task.c associated source file) performs the high-level device
00075  * application operation. Once the device is fully enumerated (device SET_CONFIGURATION request received), the task
00076  * checks for received data on its OUT endpoint and transmits this data on its IN endpoint.
00077  *   - The host template task (host_template_task.c associated file) performs the high-level host application operation.
00078  * If the B-device is correctly connected and enumerated, the task sends and receives data over the USB bus.
00079  *
00080  * \image html arch_full.jpg "Architecture Overview"
00081  *
00082  * \section config Configuration
00083  * The sample application is configured to implement both host and device functionalities.
00084  * Of course it can also be configured to be used only in device or reduced-host mode (see the conf_usb.h file).
00085  * Depending on the USB operating mode selected, the USB task will call either the USB host task (usb_host_task.c),
00086  * or the USB device task (usb_device_task.c) to manage USB specification chapter 9 requests.
00087  *
00088  * \note The B-device descriptors used for this sample application are not directly usable for enumeration
00089  *       with a standard PC host system. Please see \ref contactinfo for real device application examples
00090  *       (mass storage, HID mouse, HID keyboard, CDC, etc.).
